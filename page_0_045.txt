 Tache < 1 jour
 
Quand vous découpez en tâches (c'est à dire, découpez le travail qui doit être fait pour terminer une User Story), n'arrétez pas de découper avant que toutes les taches puissent être finies en moins d'un jour.
 
Cette règle rend plus facile la visualisation des progrès et des goulets d'étranglements. Elle déclenche aussi des discussions plus collaboratives durant les réunions de planning.
 
Si vous utilisez la pratique classique de marquage des taches d'un point pour chaque jour passé dessus, vous saurez ainsi qu'il y a quelque chose d'anormal si une des taches commence à collecter des points alors qu'aucune tache ne doit en recevoir plus d'un.
 
 Si vous utilisez le sprint burndown, vous pouvez compter les taches au lieu des heures. Plus rapide!

 Task size < 1 day

When tasking (i.e. breaking down the work that needs to be performed to complete a User Story) don’t stop splitting the tasks until each task can be completed within a day.

This policy makes it easier to see progress and bottlenecks. It also triggers deeper collaborative discussions during the planning sessions.

Furthermore, if you use Dotting, you know there is something fishy going on if a task starts collecting dots since no single task should have more than one dot.

If you use a Sprint Burndown, you can instead of summarizing hours summarize the number of tasks. Much faster!
